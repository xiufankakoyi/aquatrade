# Performance Optimization Plan for Backend

## 1. Cache Preloading Implementation

### 1.1 Trading Dates Caching
- **Issue**: Trading dates are fetched from the database every time (2.817s per request)
- **Fix**: Implement a cache for trading dates that survives across requests
- **Implementation**: 
  - Cache the results of `get_trading_dates()` in memory or a file-based cache
  - Add cache invalidation logic for when new data is added
  - Expected improvement: Reduce trading dates fetch time to near-zero

### 1.2 Frequently Accessed Data Caching
- **Issue**: No cache for stock_daily, benchmark_data, etc.
- **Fix**: Add caching for frequently accessed datasets with appropriate TTL
- **Implementation**:
  - Cache vectorized data loads for specific date ranges
  - Use a key-based caching system (e.g., `cache_key = f"stock_data_{start_date}_{end_date}"`)
  - Expected improvement: Reduce data loading time by 50-80%

## 2. Fix Full Table Scan on Limit Status

### 2.1 Apply Date Range Filter
- **Issue**: Limit_status table is being scanned without date filter (9.338s)
- **Fix**: Apply the same date range filter used for other tables
- **Implementation**:
  - Modify `load_limit_status()` function to accept and apply start/end date parameters
  - Ensure the filter is properly passed from the main data loading function
  - Expected improvement: Reduce limit_status load time from 9.3s to <1s

### 2.2 Verify Indexes
- **Issue**: LanceDB index may not be used for date filtering
- **Fix**: Ensure proper indexing on the date column of limit_status table
- **Implementation**:
  - Check if date column is indexed in LanceDB
  - Add index if missing
  - Expected improvement: Further reduce query time for date-filtered queries

## 3. Optimize Backtest Execution

### 3.1 Profile Backtest Execution
- **Issue**: Backtest execution takes 35.018s (91.9% of total time)
- **Fix**: Profile the backtest execution to identify specific slow components
- **Implementation**:
  - Add more detailed logging to the backtest engine
  - Identify which parts of the 35s are spent where
  - Focus optimization on the slowest components

### 3.2 Optimize Data Conversion
- **Issue**: To Pandas conversion is a bottleneck (0.466s)
- **Fix**: Use Polars directly instead of converting to Pandas
- **Implementation**:
  - Modify the code to work with Polars DataFrames throughout
  - Avoid unnecessary conversions between Polars and Pandas
  - Expected improvement: Reduce data conversion time by 50-70%

## 4. Optimize Data Loading Pipeline

### 4.1 Parallelize Data Loading
- **Issue**: Data loading is sequential (load daily → load info → load limit)
- **Fix**: Parallelize independent data loading operations
- **Implementation**:
  - Load daily, info, and limit data in parallel
  - Join them once all are loaded
  - Expected improvement: Reduce total data loading time by 30-50%

### 4.2 Optimize Join Operation
- **Issue**: Join operation takes 0.306s
- **Fix**: Optimize the join logic between tables
- **Implementation**:
  - Use more efficient join types
  - Ensure join columns are properly indexed
  - Expected improvement: Reduce join time by 20-40%

## Implementation Steps

1. **Fix Limit Status Full Table Scan** (Highest Priority)
   - Modify the limit_status loading code to apply date filters
   - Test with the same date range as other tables
   - Verify the fix reduces load time significantly

2. **Implement Trading Dates Cache** (High Priority)
   - Add in-memory cache for trading dates
   - Test that cache survives across requests
   - Add cache invalidation logic

3. **Optimize Data Conversion** (Medium Priority)
   - Modify code to use Polars directly
   - Remove unnecessary Pandas conversions
   - Test performance improvement

4. **Profile Backtest Execution** (Medium Priority)
   - Add detailed logging to backtest engine
   - Identify the exact slow components
   - Create optimization plan for the identified bottlenecks

5. **Implement Full Data Caching** (Low Priority)
   - Add caching for frequently accessed datasets
   - Implement cache invalidation strategies
   - Test performance improvement

## Expected Results

- **Total Backtest Time**: Reduce from 38.09s to <20s
- **Data Loading Time**: Reduce from 13.11s to <5s
- **Limit Status Load Time**: Reduce from 9.338s to <1s
- **Trading Dates Fetch Time**: Reduce from 2.817s to <0.1s

These optimizations will significantly improve the overall performance of the backtest engine, making it more responsive and capable of handling larger datasets and more frequent requests.
