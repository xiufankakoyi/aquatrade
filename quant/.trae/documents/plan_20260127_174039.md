# 前端错误处理系统增强方案

## 问题诊断
当前前端在发送爬虫请求后卡在 "Starting..." 状态，需要实现全面的错误处理机制。

## 实施计划

### 1. 全局错误边界 (Global Error Boundary)
- 添加 window.onerror 全局错误捕获器
- 添加 unhandledrejection 捕获未处理的 Promise 拒绝
- 在页面顶部显示全局错误横幅
- 记录所有未捕获错误到控制台

### 2. 优雅的 UI 降级 (Graceful UI Degradation)
- 网络错误时显示友好提示，而不是冻结
- 按钮添加加载状态和禁用状态
- 错误时自动恢复按钮可用性
- 添加重试按钮和刷新页面选项

### 3. 错误日志机制 (Error Logging)
- 实现客户端错误日志记录
- 记录时间戳、错误类型、堆栈跟踪
- 提供导出日志功能用于调试
- 在控制台输出详细错误信息

### 4. 用户恢复选项 (User Recovery Options)
- 网络错误时显示 "Retry" 按钮
- 服务器错误时提供 "Refresh Page" 选项
- 状态查询失败时允许手动刷新状态
- 添加 "Cancel" 按钮停止当前操作

### 5. 异步操作错误处理 (Async Error Handling)
- 为所有 fetch 请求添加超时控制
- 实现请求取消机制（AbortController）
- 添加重试逻辑（最多 3 次）
- 轮询失败时显示警告但继续尝试

### 6. 输入验证增强 (Input Validation)
- 实时验证日期格式
- 验证日期范围逻辑
- 提供上下文相关的错误消息
- 禁用无效输入的提交按钮

### 7. 错误场景测试 (Error Scenario Testing)
- 网络断开场景
- 服务器 500 错误
- 请求超时场景
- JSON 解析错误
- 组件渲染失败

## 文件修改
- `web/app.js` - 添加所有错误处理逻辑
- `web/index.html` - 添加错误横幅 UI
- `web/styles.css` - 添加错误样式

## 预期效果
- 任何错误都不会导致 UI 冻结
- 用户始终能看到当前状态和错误信息
- 提供明确的恢复路径
- 便于调试的详细日志
